Web Content Accessibility Guidelines (WCAG)

Your kung fu journey starts here. This sacred text is internationally recognized as the standard for creating web accessible websites. If you don't know WCAG or atleast how to read it, you're pretty much screwed. So let's begin!

Four Principles

WCAG is made up of 4 principles which makes up the acronym POUR. Bruce Lee referenced it numerous times when he gave his famous quote:

"You must be shapeless, formless, like water. When you pour water in a cup, it becomes the cup. When you pour water in a bottle, it becomes the bottle. When you pour water in a teapot, it becomes the teapot. Water can drip and it can crash. Become like water my friend."
I know... profound! This nugget of wisdom is going to help you remember the four principles to WCAG.

  1. Perceivable: Will a user be able to perceieve the information that you're showing them? Will they be able to perceive what you're asking them to do, and how to navigate through your site?
  2. Operable: Will a user who has limited mobility be able to operate your website with only a keyboard?
  3. Understandable: Will the user be able to understand the information and how to use the interface?
  4. Robust: Is what you've created adaptable with technology?

Principle 1: Perceivable

Information and user interface components must be presentable to users in ways they can perceive.

  • Text Alternative: Provide text alternatives for any non-text content so that it can be changed into other forms people need, such as large print, braille, speech, symbols or simpler language. (WCAG 1.1)
  • Turn on your Screen Reader (or ChromeVox) and tab to the input field. Notice how it reads the content below the text field? That's the power of Accessible Rich Internet Applications (ARIA). In the sample code below you'll see that I used the aria-describedby attribute and referenced the id="sample_input_text" containing the content.

HTML UI Components

My first rule for UI components is to always use native components first. That is, unless your company is dying to brand their own fancy schmancy components. In that case, DO NOT DESIGN or BUILD components without first understanding how native HTML UI components function on a mouse, keyboard, and screen reader. Once you do that, move onto learning what Accessible Rich Internet Applications (ARIA) attributes, roles, or states are needed to make your component work properly for assistive technologies.

Input Text Fields

It's hard to get this wrong folks. It comes great out of the box and you style it however you want to (within reason). Now here are a fewmnthings to keep in mind:

  • Always use a label. It's required for meeting the Web Content Accessibility Guidelines. (WCAG 3.3.2.)
  • LEAVE the glorious focus state ALONE... unless you have a badass plan for "enhancing" it... but if you're reading this, you probably don't. So leave it alone! (WCAG 2.4.7)
  • Placeholder text should never be used in place of a label. It's contrast ratio is too low, and it's not read by some screen readers. Use it, but don't EVER use it in place of a label. (WCAG 1.4.3)

Key Features:

  • Click on the label and notice how the text field receives focus!
  • Turn on your Screen Reader (or ChromeVox) and tab to the input field. Notice how it reads the content below the text field? That's the power of Accessible Rich Internet Applications (ARIA). Select Dropdowns

Alright, the select dropdowns can be ugly AF. But by adding -webkit-appearance: none; you can have a bit of customization over the select field. However, if you're going to go native (recommended), you'll have to bear with the native dropdown appearance (it's not soooo bad). Now if you or your company thinks it's THAT bad, you may want to simply create your own dropdowns. But remember, observe the functionality of the native select dropdown first before you go off creating your own!

  • Always use a label. It's required for meeting the Web Content Accessibility Guidelines. (WCAG 3.3.2.)
  • LEAVE the glorious focus state ALONE! (WCAG 2.4.7)
  • Placeholder text should never be used in place of a label.
  • To set up a placeholder text for select dropdowns, you must set option to disabled and selected
    • (WCAG 1.4.3)

Key Features:

  • Click on the label and notice how the select field receives focus!
  • Notice the beautiful contrast as you hover over the dropdown options
  • Use tab to bring the select field into focus. When you hit tab again, the focus goes to another ui component.
  • They keyboard arrow keys enable you to select options from the dropdown.
  • Turn on your Screen Reader (or ChromeVox) and tab to the input field. Notice how it reads the content below the text field? That's the power of Accessible Rich Internet Applications (ARIA). In the sample code below you'll see that I used the aria-describedby attribute and referenced the id="sample_select_text" containing the content.

<label for="native-select">Label for Select></label>
<select name="" id="native-select" aria-describedby="sample_submit_text" required>
    <option value="" disabled selected> Select Placeholder</option>
    <option value="1">Batman</option>
    <option value="2">Robin</option>
    <option value="3">Night Wing</option>
    <option value="4">Crazy Robin</option>
</select>
